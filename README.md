# sky-demand-evaluator

`sky-demand-evaluator` 是一个用于「评估产品需求真伪 + 给出实施优先级 + 辅助产出产品化描述」的 Codex Skill。

它主要解决这样一个场景：  
当用户、同事、运营或老板提出一条「需求 / 吐槽 / 功能想法」时，如何把这句话系统地拆解成：

- 这是不是一个真需求（而非伪需求或仅是方案偏好）；  
- 值不值得做、什么时候做（实施推荐度评分、价值 vs 成本）；  
- 对应的用户故事、关键使用路径；  
- 如果值得做，进一步收敛成更具体的产品方案建议。

## 主要能力概览

当提供一段需求或痛点描述时（例如：  
「APP 首页应该少一些初始化弹框」  
「首页金刚区按钮不要显示图片只显示文字」  
「广告要增加关闭按钮」等），  
`sky-demand-evaluator` 会引导 Codex 输出一个结构化的分析结果，包括但不限于：

1. **需求结论标签**：真需求 / 伪需求 / 待验证。  
2. **需求类型标签**：新功能、体验优化、商业化、运营配置、风险合规、技术底座等。  
3. **实施推荐度评分（0%–100%）**：当前是否推荐实施以及理由。  
4. **背后问题与用户目标**：从「方案」还原到真实问题和目标。  
5. **关键判断依据**：从价值、频率、范围、可替代性、一致性五个维度解释判断。  
6. **价值 vs 成本评估**：分别给出价值档位与实施成本档位。  
7. **建议的产品化描述**：用户故事 / Job story + 4–7 步的关键用户使用路径。  
8. **数据与埋点建议**：要看哪些数据、如何补埋点、这些数据用来回答什么问题。  
9. **典型反模式提醒**：如只提方案不提问题、纯审美偏好等，帮助识别伪需求。  
10. **（满足条件时）具体产品方案建议**：  
    - 当结论为真需求且实施推荐度 > 50% 时，Skill 要求额外输出一份简要产品方案，包括：  
      - 功能目标与成功标准；  
      - 主要人群与场景；  
      - 核心交互流程与关键态；  
      - 约束和防滥用策略；  
      - 需要后续确认的开放问题。

## 何时使用这个 Skill

`SKILL.md` 中列出了一些典型触发场景，简要总结如下：

- 当有人提到**具体功能/按钮/配置**：  
  - 例：「首页应该少一些初始化弹框」「这里要多加一个筛选项」「按钮放大一点」。  
- 当你从**群聊、工单、评论、调研**中截取到一段抱怨或建议，想判断是不是值得立项的需求。  
- 当你在做**需求池梳理 / 评审 / 排期**时，想用统一口径评估某条需求。  
- 当你希望为版本规划提供「价值 vs 成本 + 实施推荐度」的辅助判断。

一句话：  
只要你在想「这个要不要做」「值不值得优先做」「到底算不算真需求」，都可以调用这个 Skill。

## 在 Codex 中如何使用

前提：你的 Codex 环境可以识别本地 Skill，并将 `skills/sky-demand-evaluator` 所在路径加入到 Skill 搜索路径中（通常是拷贝到 `$CODEX_HOME/skills` 或在配置中指明该目录）。

1. **安装 / 拷贝**  
   - 将本仓库中的 `skills/sky-demand-evaluator` 目录拷贝到你的 Codex Skill 目录，例如：  
     - `cp -R skills/sky-demand-evaluator ~/.codex/skills/`

2. **触发使用**  
   - 在对话中，你可以通过名称显式指出希望用此 Skill 协助，例如：  
     - 「请用 `sky-demand-evaluator` 帮我评估这条需求：……」  
   - 或者在系统/开发者级配置中，为该 Skill 的 description 配置自动触发规则（如果支持）。

3. **输入格式建议**  
   - 建议提供：  
     - 原始用户的话（可包含截图转写、工单内容、聊天记录等）；  
     - 已知的用户画像/场景（如果有）；  
     - 任何相关的业务背景（例如当前版本目标、已有数据结论）。  
   - Skill 会在信息不足时，提示需要补充哪些信息，或在输出中标注合理假设。

## 示例（简化版）

- 输入（示例）：  
  - 「用户说：APP 首页应该少一些初始化弹框。」

- 输出（结构要点）：  
  - 需求结论标签：真需求（体验类）  
  - 需求类型标签：体验优化 / 商业化相关  
  - 实施推荐度评分：70%  
  - 背后问题与用户目标：用户希望更快看到核心内容，不被多次弹窗打断。  
  - 关键判断依据：从价值/频率/范围/可替代性/一致性几个维度说明。  
  - 价值 vs 成本：价值偏高、成本中等。  
  - 用户故事 + 关键用户路径：  
    - 用户故事示例：  
      - 「作为一名新用户，当我第一次打开 APP 首页时，我希望不被多次弹窗打断，从而可以快速理解这个产品能为我做什么。」  
    - 用户路径示例：  
      - 「首次打开 APP → 看到首页主要内容 →（必要时）出现 1 个关键引导弹窗 → 完成或关闭 → 继续浏览首页 / 进入核心功能。」  
  - 数据与埋点建议：  
    - 查看首页跳出率、弹窗展示/关闭率、弹窗后的继续行为；  
    - 如埋点不足，建议新增「首页曝光」「弹窗展示/关闭」等事件。  
  - （满足条件）具体产品方案建议简要轮廓。

完整、详细的结构和说明请参考 `skills/sky-demand-evaluator/SKILL.md`。

## 开发和扩展建议

- 如果你的团队有固定的指标体系（例如特定的漏斗定义、核心事件、业务线分类），可以基于当前 SKILL.md：  
  - 增加自定义的「需求类型标签」枚举；  
  - 把「价值 vs 成本」的判定规则和阈值写得更贴近你们的实践；  
  - 在「数据与埋点建议」部分补充你们的标准埋点规范。  
- 推荐的迭代方式：  
  1. 用本 Skill 评估一批真实需求；  
  2. 总结哪些信息还经常要在对话里补充；  
  3. 回写到 SKILL.md 的说明里，让下一次评估更顺畅。
