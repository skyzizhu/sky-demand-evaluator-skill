# 使用说明

本 skill 用于把「用户说的一段需求 / 吐槽 / 功能想法」，转化为结构化的需求判断与产品化表达，并给出一个实施推荐度评分，帮助判断这是「真需求」「伪需求」还是「待验证」，以及当前是否值得优先实施。  
当结论为「真需求」且实施推荐度评分 > 50% 时，还需要在输出中补充一份简要的「具体产品方案建议」，将口头需求转换成相对可落地的产品需求描述。

## 0. 何时使用本 skill

在出现以下情况时，优先考虑调用 sky-demand-evaluator：

- 有用户或同事提出某个具体功能 / 按钮 / 配置方案，例如：「APP 首页应该少一些初始化弹框」「这里要多加一个筛选项」「能不能把这个按钮放到更显眼的地方」。  
- 从群聊、工单、用户调研、评论里截取到一段抱怨或建议，希望判断它是不是值得立项的真需求。  
- 在做需求池梳理、排期或需求评审时，某一条需求存在争议，希望用统一口径做结构化分析和打分。  
- 在做版本规划时，需要比较几条不同需求的价值和优先级，输出相对客观的实施推荐度。  

只要用户提供了一段需求或痛点描述，并且你在想「要不要做」「优先级有多高」，就可以使用本 skill。

## 1. 接收输入并提炼关键信息

当用户提供一段需求、吐槽或功能建议时（包括聊天记录、工单、调研回答、邮件等），按以下步骤处理：

- 识别基本要素（如果上下文信息不足，则根据文本合理推断）：
  - 用户角色 / 画像（例如：新手用户、重度用户、渠道代理、运营同学等）
  - 使用场景（在什么情境、什么任务下提出这个需求）
  - 当前行为和路径（现在是如何完成这件事的，大致步骤是什么）
  - 主要痛点或目标（不舒服的地方 / 想达成的结果）
  - 粗略的发生频率（经常遇到 / 偶尔遇到 / 第一次）
  - 影响范围（只影响个人、少数用户，还是一类核心用户）

- 如关键信息明显缺失，可以向提问者补充 3–5 个简短问题，问题要高度聚焦在：
  - 这个情况多久遇到一次？
  - 不解决会有什么影响？
  - 现在是怎么绕过去或解决的？
  - 你最希望变成什么样？

如果无法补问（例如只提供了一段历史记录），则基于文本内容做清晰、保守的推断，并在输出中标注「假设」和「不确定点」。

## 2. 使用真伪需求判断框架

从以下 5 个维度，对这条需求进行判断，并为每个维度输出一句话说明：

- 价值（Impact）  
  - 问题对用户结果的影响有多大？是影响结果成败，还是只影响体验舒适度。

- 频率（Frequency）  
  - 这是高频场景还是偶发场景？用户大致多久会遇到一次。

- 范围（Coverage）  
  - 影响的是一类核心用户，还是少数极端/边缘用户。

- 可替代性（Substitutability）  
  - 现在是否有还算可用的替代方案？替代方案的成本和痛苦有多大。

- 一致性（Alignment）  
  - 该需求是否符合产品当前的方向和阶段目标（例如当前更关注留存、转化、营收、规模等）。
  - 如果缺少战略/阶段信息，可根据对产品类型的常识假设，并在输出中标注这是假设。

在综合以上维度后，将需求归类为：

- 「真需求」：高频或高强度，影响一类核心用户，对结果有实质影响，且现有方案难以替代，同时与产品方向大体一致。
- 「伪需求」：更多是某种具体方案偏好、个别用户习惯或边缘场景，对结果影响有限，且容易被现有功能或简单改动替代。
- 「待验证」：关键信息不足，或高度依赖数据/进一步用户研究才能确认。

## 3. 输出结构化分析结果

无论最终判断如何，都按以下统一结构输出结果，方便产品决策和后续沟通：

1. 需求结论标签  
   - 在「真需求 / 伪需求 / 待验证」中选择一个，并用 1 句话总结原因。

1.1 需求类型标签  
   - 为当前需求打上 1–2 个类型标签，用于后续排期和沟通，例如（可多选）：  
     - 新功能 / 能力补齐  
     - 体验优化 / 易用性提升  
     - 风险与合规 / 安全  
     - 商业化相关（广告、付费、转化漏斗）  
     - 运营配置 / 策略调优  
     - 技术底座 / 性能 / 稳定性（如文中有明显线索时才使用）  
   - 如无法明确归类，可用「待归类」但要说明原因。

2. 实施推荐度评分  
   - 在 0%–100% 之间给出一个「当前推荐实施」的主观评分，100% 表示强烈推荐立刻实施，0% 表示当前完全不推荐实施。  
   - 评分应综合前述五个维度（价值、频率、范围、可替代性、一致性）以及信息充分度：  
     - 一般参考口径（可根据实际情况微调）：  
       - 80%–100%：高价值 + 高频/高强度 + 覆盖核心用户 + 难以被替代 + 与当前目标高度一致。  
       - 40%–80%：有一定价值或覆盖，但在频率、范围、可替代性或方向一致性上存在明显 trade-off，需要结合资源和节奏权衡。  
       - 0%–40%：价值有限、覆盖范围小、已有不错的替代方案，或与当前产品方向不一致，暂不建议实施。  
   - 如信息明显不足，应适当降低分值并在说明中标注「由于信息不足，该评分仅供初步参考」。

3. 背后问题与用户目标  
   - 用 1–2 句话，描述用户真正想解决的问题或达成的目标，避免只重复功能表述。  
   - 可以使用「用户现在在【场景】，为了【目标】，被【具体障碍】卡住」的句式。

4. 关键判断依据（按维度拆分）  
   - 价值：……  
   - 频率：……  
   - 范围：……  
   - 可替代性：……  
   - 一致性：……  
   每个维度用简短自然语言说明，必要时标注假设前提。

4.1 价值 vs 成本评估  
   - 在上述维度基础上，单独给出：  
     - 预估价值档位：高 / 中 / 低（从对核心指标的潜在影响角度评估，例如留存、转化、营收、口碑等）。  
     - 预估实施成本档位：高 / 中 / 低（综合考虑产品、设计、开发、测试、联调、运营配置等投入）。  
   - 如信息不足，允许用「不确定（倾向高/中/低）」的表达，并说明主要的不确定因素。  
   - 在解释实施推荐度评分时，应明确它是基于「价值 vs 成本」和风险权衡后的综合结果。

5. 建议的产品化描述  
   - 先用 1–2 条「用户故事」或「Job story」形式描述需求：  
     - 用户故事示例：  
       - 「作为一名【用户角色】，当我在【场景】时，我希望【行为/能力】，从而【达成的结果】。」  
     - Job story 示例：  
       - 「当我在【情境】时，我想要【动机 / 行为】，从而可以【预期结果】。」  
   - 在此基础上，补充一条简短的关键使用路径（用户路径），用 4–7 个节点描述用户完成该任务的主要步骤，例如：  
     - 「打开 APP 首页 → 浏览金刚区入口 → 识别到【某功能入口】 → 点击进入功能页 → 完成【关键操作】 → 查看结果/反馈」。  
   - 节点应尽量对应到后续可能设计或埋点的功能节点（如页面、按钮、关键操作），但保持简洁，不做过度细化。

6. 验证与下一步建议  
   - 需要优先查看或补充的数据点（例如：某步漏斗转化、该路径的使用频率、异常率等），并说明这些数据各自要回答什么问题。  
   - 如现有埋点体系不足，给出 1–3 条具体「埋点或日志采集建议」，包括：  
     - 事件名称 / 节点位置（例如：点击某按钮、进入某页面、完成某操作）；  
     - 建议采集的关键属性（例如：用户类型、入口来源、设备信息、业务关键参数等）；  
     - 预期通过这些埋点观察到的现象（例如：是否真有大量用户在某一步流失）。  
   - 可以快速执行的验证方式（例如：小范围灰度、AB 测试、和 3–5 位目标用户的半结构化访谈），并简要给出实验/访谈的大致设计思路。  
   - 如果是「待验证」，明确写出建议先做哪些验证，再回到是否要立项和排期的判断。

## 4. 使用口径与风格约定

在使用本 skill 时，请遵循以下口径：

- 面向产品/运营同学输出，语言可以专业但要清晰、克制，避免空泛的大词。  
- 对自己的推断要标注哪些是「事实」，哪些是「合理假设」，避免把猜测说成结论。  
- 即使判断为「伪需求」，也要尊重用户感受，用分析说明「为什么优先级可能不高」，而不是简单否定。

当用户没有特别说明时，默认：

- 产品目标偏向「提升核心留存和关键转化」，你可以在一致性分析中基于这一点做推断并说明。  
- 分析的优先级排序为：先看价值和频率，再看范围和可替代性，最后看与产品方向的一致性。

## 5. 输出格式示例（仅供参考）

在实际回答中不必逐字照搬此示例，但建议保持类似的结构和信息密度：

- 需求结论标签：真需求（理由：……）  
- 实施推荐度评分：80%（理由：……）  
- 背后问题与用户目标：……  
- 关键判断依据：  
  - 价值：……  
  - 频率：……  
  - 范围：……  
  - 可替代性：……  
  - 一致性：……  
- 建议的产品化描述：  
  - 用户故事：……  
- 验证与下一步建议：……  
- （如满足条件）具体产品方案建议：……  

其中，「具体产品方案建议」仅在满足以下条件时输出：

- 需求被判断为「真需求」；且  
- 实施推荐度评分 > 50%。  

在这种情况下，产品方案建议应尽量包含：

- 功能目标与成功标准（要解决什么、做到什么程度算成功）；  
- 主要使用人群和场景；  
- 核心交互流程和关键态（例如：何时出现、何时消失、按钮文案与位置等）；  
- 基本约束和防滥用策略（例如频次控制、策略例外）；  
- 仍然需要后续讨论或验证的开放问题。

只要用户提供了一段需求或痛点描述，就使用以上流程进行判断和输出；如满足条件，则在结尾追加上述结构化的产品方案建议。

## 6. 常见反模式与注意事项（可作为使用者提醒）

在使用本 skill 时，如遇到以下类型的描述，要特别注意它们很可能属于「伪需求」或「信息不足」场景：

- 只给出具体方案，不说明问题  
  - 例如：「这里加个导出按钮」「把这个弹窗关掉」；  
  - 处理方式：尽量追问或自行补全「为什么」「不加会怎样」。

- 只代表极少数内部角色的偏好  
  - 例如：「运营觉得这个入口要放大」「老板觉得颜色太丑」；  
  - 处理方式：分析对真实终端用户和业务指标的影响，而不是以个人审美为主。

- 纯视觉风格诉求，没有功能或效率相关的实质问题  
  - 例如：「看起来太土了」「整体要高级一点」；  
  - 处理方式：可收录为设计改版参考，但在本 skill 下优先评估其对任务完成和指标的实际影响。

- 过于宽泛的方向性口号  
  - 例如：「要提高用户活跃度」「让首页更智能」；  
  - 处理方式：拆成更具体的场景+任务+行为，再进行评估。

遇到上述情况，输出中应明确指出：目前信息不足以判断为强烈推荐实施的具体需求，并列出需要补充的信息或拆解方向。

## 7. 输入与输出示例（节选）

为了便于理解，本节给出一个简化示例。实际使用中不必逐字照搬结构，但建议保持类似信息密度。

- 示例输入（来自用户反馈或群聊）：  
  - 「APP 首页应该少一些初始化弹框。」  

- 示例输出要点（简化版）：  
  - 需求结论标签：真需求（理由：初始化弹框频繁打断首页主流程，影响体验和早期留存）  
  - 需求类型标签：体验优化 / 商业化相关  
  - 实施推荐度评分：70%（理由略）  
  - 背后问题与用户目标：用户想更快进入首页核心内容，减少被打断感。  
  - 关键判断依据：按 5 个维度分别说明。  
  - 价值 vs 成本评估：价值偏高，成本中等。  
  - 建议的产品化描述：  
    - 用户故事：  
      - 「作为一名新用户，当我第一次打开 APP 首页时，我希望不被多次弹窗打断，从而可以快速理解这个产品能为我做什么。」  
    - 关键使用路径：  
      - 「首次打开 APP → 看到首页主要内容 →（必要时）出现 1 个关键引导或授权弹窗 → 完成或关闭 → 继续浏览首页 / 进入核心功能。」  
  - 验证与下一步建议：  
    - 查看首页首屏的跳出率与弹窗曝光次数关系；  
    - 如埋点不足，建议新增「首页曝光」「弹窗展示/关闭」「弹窗后继续行为」等事件；  
    - 设计减少弹窗数量或改为轻提示的 AB 实验方案。  
  - （如满足条件）具体产品方案建议：  
    - 例如：「首页初始化弹窗精简方案 v1」，简要列出只保留哪 1–2 个关键弹窗、出现时机和频次限制等。

你可以参考该示例，按同样结构分析自己的真实需求场景。
